Specification

Web Architecture

We created a clear separation between the presentation layer and the application layer.

  • The Presentation layer is handled by the browser that requests page assets, sends HTTP requests (also AJAX calls), and then renders the results on the client-side.
  • The Application layer, that is in the backend server, is responsible of the communication with the data layer to perform request on NoSQL DB and exposed it in form of REST API.

    • The js frontend invokes this API to get the data needed in the pages and to perform actions on it to serve dynamically the user requested page.

API

REST compliance

We respect the REST principles to create an API that allows client and server to grow independently, and that provides a uniform interface where everything related to event is under the path /event and same goes for artist, seminar, user and cart. We use REST principles for authentication requests too, soo we implement stateless server about client information (Cookie), usign a JSON Web Token, saving this client-side and adding this in the Authorization Header for authenticated request to server.

OpenAPI Resource models

The main entities involved in our festival web site are Artist, Event, Seminar, User and Order. Artist, Event and Seminar contain the attributes requested in the specification by the instructors. The event also contains a list of Artist’ IDs to quickly access the Artist information, and Seminar'IDs. For each of them we created a resource model. In these models all the properties are mandatory, so we first create artists and seminars and then with their id's we populate events

  • We first create Artist and Seminar that have no relation in their table
  • Second we created Event's table with ID reference from the previus created
  • For registration user send Name, Surname, Email and password (the last in BASE64 encode) for register.
  • For login, if the request was accepted by server, we response with JWT token that saved in localStorage of the Browser.

All single path have specific request, and Resource model, params and body to use for correct request and interation.Their documentation can be found in API docs (D1 or D2).

Note about POST method for Populate DB

For semplicity, we authorize root user same as normal client user. So the post method for /artist /seminar and /event are accesible for all user authenticated. We don't introduct double level of security because is not the main topic of this course, but we are informated that is not the best practice in web site development.

Implementation

Tools used

We developed our NoSQL database with MongoDb Atlas for cloud integration. We use POSTMAN tool for interact with API, for initial testing and for populate DB throght the POST path. The integration with the MongoDB database was made possible by the Mongoose.js dependencies, that permit to create and manage a cloud-based database. We our DB passsword in heroku process.env for security reason. As Code editor we all used VSCode and Chrome as a browser using their console for debug and test.

We used the framework Express.js to handle the routing of all the deliverables, including this one. Express.js perform all server funtionality in integration with other depencendencis for utils such as body JSON parser, JWT token for authentication and MULTER for photo gallery uploads.

    For our backend side we use this js dependencies:

  • "@fullcalendar/daygrid
  • bcrypt3
  • body-parser
  • express
  • jsonwebtoken
  • mongoose
  • morgan
  • multer

For implementation of calendar, because Boostarp not serve native, we used FullCalendar.js plugin. Here is their documentation.

We use GitHub as version control support. We used Heroku to deploy our applications in the cloud platform, where we setup a continuous delivery mechanism linked with GitHub master branch. We also used Swagger for create OpenAPI documentation in YAML format and generate SwaggerUI.

Discussion

The backend server that is the only one interfacing the DB and that provides a RESTful API to operate on it, the frontend application client request REST resources, show pages and invoke the APIs in response to user interaction with the page. In the frontend there is a separation between the provided pages and static assets like stylesheets, scripts and static images.

For Session state we choose for JWT for their semplicity and for respect REST API principles, the statless server, so when user perform login request, server not store any client information, but they generate a JWT token. This one will be store in localSotrage of the browser and used client side for authenticated request such as cart and order funtionality. Information about user will deleted client side when customer close the broswer or when they clic Log Out. Server will accept token for 12h and then though the client don't close browser they will be logged out.
